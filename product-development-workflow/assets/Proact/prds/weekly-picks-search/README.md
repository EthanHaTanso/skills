## 1. Product 목표 & Success 정의

### 1.1. 제품 컨텍스트 요약

- **Proact**의 핵심 가치:
    > "매주 내 비즈니스에 딱 맞는 정부 입찰 공고를 자동으로 추천받아, 새로운 영업 기회(Sequence)를 놓치지 않는다"

- **지금까지:**
    - (1) **온보딩 공고 선택**: 유저가 관심 공고를 선택하여 비즈니스 프로필 생성
    - (2) **Relevant Score 계산**: Top 10 keywords 기반 Vector 유사도로 공고 추천 점수 산출
    - (3) **Weekly Picks**: 매주 새 공고를 Relevant Score 순으로 정렬하여 제공
    - (4) **NAICS/PSC 필터**: 산업 분류 코드 기반 필터링 (Default: Relevant Score 상위 10개)

- **새로 추가하는 키워드 검색 기능:**
    - (5) **Weekly Picks 내 키워드 검색**: 유저가 직접 키워드를 입력하여 이번 주 새 공고 중에서 원하는 공고를 추가로 탐색

---

### 1.2. 왜 키워드 검색이 중요한가 (Aha-Moment 관점)

- **현재 제품의 Aha-moment**는 "Weekly Picks에서 내 비즈니스에 맞는 공고를 발견"에 과도하게 의존
    - **문제점**: Relevant Score 로직(Top 10 keywords 기반)의 한계로 인해, 실제로 적합한 공고가 상위 70% 범위 밖에 묻히는 경우 발생
    - **결과**: 약 10% 유저가 "이번주 할 수 있는 공고가 없다"고 판단하고 이탈
    - **임팩트**: Weekly Picks = Sequence 생성 트리거 → 트리거 실패 시 제품 이탈

- **우리가 통제 가능한 Aha:**
    - "상위 공고에서 못 찾았지만, 키워드로 검색했더니 딱 맞는 공고를 발견했다"
    - 이 경험이 "Proact은 내가 놓칠 뻔한 기회도 찾아준다"는 신뢰로 이어짐

> **따라서 키워드 검색의 최우선 목표:**
> - **"Weekly Picks 이탈률 10% → 5% 이하로 감소"**
> - **"검색을 통한 Sequence 생성 수 주간 50건 이상 달성"** (전체 Sequence 생성의 약 10% 목표)

## 3. 유저 가치 & 비즈니스 가치 정리

### 유저 가치

1. **탐색 실패로 인한 기회 손실 방지**
   - Relevant Score 로직이 놓친 공고를 키워드 검색으로 발견 가능
   - "혹시 모를 기회"를 확인할 수 있는 안전망 제공

2. **탐색 효율성 향상**
   - 특정 키워드 공고를 빠르게 찾을 수 있음
   - 전체 리스트 스크롤 없이 타겟팅된 탐색 가능

3. **명확한 종료 경험**
   - "이번 주는 없다"는 확신을 갖고 깔끔하게 다음 주로 넘어갈 수 있음
   - 불필요한 시간 낭비 및 좌절감 방지

### 비즈니스 가치

1. **Weekly Picks 리텐션 개선**
   - 현재 10% 이탈 → 5% 이하로 감소 목표
   - 유저가 "Weekly Picks = 기회 발견 도구"로 신뢰 강화

2. **Sequence 생성 수 증가**
   - 검색을 통한 Sequence 생성 주간 50건 이상 목표
   - 전체 Sequence 생성률 약 10% 증가 기대

3. **제품 차별화 요소 확보**
   - Relevant Score 로직이 완벽하지 않아도 유저가 직접 보완 가능
   - "AI 추천 + 사람의 직관" 조합으로 경쟁력 확보

## 4. 키워드 검색 기능 전체 유저 플로우 (High-level)

### Flow A. 검색창 노출 및 검색 성공 (기본 플로우)

- **Entry Point**: Weekly Picks 페이지
- **진입 트리거**: 유저가 상위 10개 공고까지 스크롤
- **진행 과정**:
  - Step 1: 검색창 자동 노출 (Progressive Disclosure)
  - Step 2: 유저가 키워드 입력 및 검색
  - Step 3: 매칭된 공고 리스트 필터링 표시 (Relevant Score 순서 유지)
  - Step 4: 유저가 적합한 공고 발견 → 클릭하여 상세 확인
- **완료 후**: Sequence 생성으로 이어지거나, 다른 키워드로 재검색

---

### Flow B. 검색 결과 없음 (1차 시도)

- **Entry Point**: 검색창에 키워드 입력
- **진입 트리거**: 유저가 검색 실행
- **진행 과정**:
  - Step 1: 검색 실행
  - Step 2: 매칭 결과 0건
  - Step 3: "결과가 없습니다. 다른 키워드로 검색해보세요" 메시지 표시
  - Step 4: 유저가 다른 키워드로 재검색 시도
- **완료 후**: 재검색 또는 포기

---

### Flow C. 검색 결과 없음 (2차 시도)

- **Entry Point**: 1차 검색 실패 후 재검색
- **진입 트리거**: 유저가 두 번째 검색 실행
- **진행 과정**:
  - Step 1: 재검색 실행
  - Step 2: 매칭 결과 0건 (또 다시)
  - Step 3: "결과가 없습니다. 이번 주는 관련 공고가 없는 것 같아요. 다음 주에 다시 확인해보세요" 메시지 표시
- **완료 후**: 유저가 납득하고 Weekly Picks 종료, 다음 주 대기

---

## 5. 상세 유저 플로우

### 5.1. 검색창 자동 노출 — 탐색 확장 옵션 제공

**상황**

- 유저가 Weekly Picks 페이지에서 상위 10개 공고까지 스크롤했을 때
- 유저의 심리/기대: "상위 공고에서 못 찾았는데, 혹시 더 아래에 있을까? 근데 끝까지 스크롤하기는 귀찮은데..."

**플로우**

1. 시스템이 유저의 스크롤 위치를 감지 (상위 10개 도달)
2. 공고 리스트 상단에 검색창 자동 노출 (Sticky header 또는 In-list CTA 형태)
3. Placeholder 텍스트 표시: "관심 기술이나 서비스 키워드를 검색해보세요"

**디자인 의도**

- **Progressive Disclosure**: 초기에는 복잡도를 낮추고, 필요할 때만 검색 기능 노출
- **Just-in-time 제안**: 유저가 "못 찾았네"라고 느끼는 시점에 정확히 대안 제공
- **낮은 진입 장벽**: 검색창만 노출하고, 사용은 유저 선택에 맡김 (강요하지 않음)

---

### 5.2. 키워드 입력 및 검색 — 명확한 Input/Output

**상황**

- 검색창이 노출된 상태에서 유저가 특정 키워드를 떠올렸을 때
- 유저의 심리/기대: "Cloud Migration 관련 공고가 있는지 확인해보고 싶다. 있으면 좋고, 없으면 다른 키워드 시도해보면 되지"

**플로우**

1. 유저가 검색창에 키워드 입력 (예: "Cloud Migration")
2. Enter 또는 검색 버튼 클릭
3. 시스템이 이번 주 새 공고의 Title + Description에서 해당 키워드 포함 공고 검색
4. 매칭된 공고만 필터링하여 리스트 업데이트 (Relevant Score 순서 유지)

**디자인 의도**

- **명확한 노력의 상한**: 유저는 "키워드 하나 입력 = 1회 시도"로 인식, 부담 없음
- **즉각적인 피드백**: 검색 결과를 바로 리스트에 반영하여 대기 시간 최소화
- **기존 맥락 유지**: Relevant Score 순서를 유지하여 "가장 관련도 높은 것부터" 보는 경험 유지

---

### 5.3. 검색 결과 확인 — 발견 또는 재시도

**상황**

- 검색 결과가 표시된 상태
- **Case A**: 매칭된 공고 1개 이상 (성공)
- **Case B**: 매칭된 공고 0개 (실패)

**플로우 - Case A (성공)**

1. 필터링된 공고 리스트 표시 (예: 5개 매칭)
2. 유저가 공고 클릭하여 Description 확인
3. 적합한 공고 발견 시 → "Save to Sequence" 또는 상세 페이지 이동

**플로우 - Case B (1차 실패)**

1. "결과가 없습니다. 다른 키워드로 검색해보세요" 메시지 표시
2. 검색창은 그대로 유지 (재입력 가능)
3. 유저가 다른 키워드로 재시도 또는 검색 종료

**플로우 - Case C (2차 실패)**

1. "결과가 없습니다. 이번 주는 관련 공고가 없는 것 같아요. 다음 주에 다시 확인해보세요" 메시지 표시
2. 유저가 납득하고 Weekly Picks 종료

**디자인 의도**

- **Case A**: 발견의 기쁨 → 즉시 행동으로 연결 (Sequence 생성)
- **Case B**: 1회 실패는 자연스러운 탐색 과정으로 수용, 재시도 유도
- **Case C**: 2회 실패는 "진짜 없구나" 신호로 인식, 깔끔한 종료 경험 제공
- **핵심**: 유저가 "충분히 시도했다"고 느끼고 납득할 수 있게 함

---

### 5.4. 검색 취소 및 원래 리스트 복귀 — 자유로운 탐색

**상황**

- 검색 결과를 확인했지만, 원래의 전체 리스트로 돌아가고 싶을 때
- 유저의 심리/기대: "검색 결과는 별로였네. 다시 전체 리스트를 보고 싶다"

**플로우**

1. 검색창 옆에 "Clear" 또는 "X" 버튼 클릭
2. 필터가 해제되고 원래의 전체 공고 리스트로 복귀 (Relevant Score 순)
3. 검색창은 그대로 유지 (다시 검색 가능)

**디자인 의도**

- **자유로운 탐색**: 검색 → 취소 → 재검색 반복 가능
- **실수 복구**: 잘못된 키워드 입력 시 쉽게 되돌릴 수 있음
- **탐색 연속성**: 검색과 스크롤 탐색을 자유롭게 오갈 수 있음

## 6. 기술 사양

### 6.1. 검색 로직

**검색 범위:**
- 이번 주 새로 업데이트된 공고 (Weekly Picks에 이미 표시된 공고)
- 전체 DB는 제외 (Weekly Picks = 이번 주 트리거 역할)

**검색 대상 필드:**
- `title` (공고 제목)
- `description` (Background & Objective + Scope of Work 전체 텍스트)

**검색 방식:**
- 기본: Case-insensitive 부분 일치 (PostgreSQL `ILIKE` 또는 MySQL `LIKE`)
- 향후 개선: Full-text search (PostgreSQL `ts_vector`, Elasticsearch 등)

**결과 정렬:**
- Relevant Score 기준 내림차순 (기존 순서 유지)

---

### 6.2. 프론트엔드 동작

**검색창 노출 조건:**
- 유저가 상위 10개 공고까지 스크롤
- Intersection Observer API 또는 스크롤 이벤트 리스너 사용

**검색창 UI:**
- 위치: 리스트 상단 Sticky header 또는 In-list CTA
- 요소:
  - Input 필드: Placeholder "관심 기술이나 서비스 키워드를 검색해보세요"
  - 검색 버튼: "Search" 또는 돋보기 아이콘
  - Clear 버튼: "X" 아이콘 (검색 후 노출)

**상태 관리:**
- 검색 전: 전체 공고 리스트
- 검색 중: Loading indicator (선택적)
- 검색 후: 필터링된 공고 리스트 + 결과 수 표시 (예: "3건의 공고를 찾았습니다")
- 결과 없음: 메시지 표시 (1차/2차 구분)

---

### 6.3. 백엔드 API

**Endpoint:**
```
GET /api/weekly-picks/search?keyword={keyword}&week={week_id}
```

**Parameters:**
- `keyword`: 검색 키워드 (string, required)
- `week_id`: 이번 주 식별자 (string, required)

**Response:**
```json
{
  "total": 3,
  "opportunities": [
    {
      "id": "opp_12345",
      "title": "Cloud Infrastructure Modernization",
      "description": "...",
      "relevant_score": 0.87,
      "naics_codes": ["541512"],
      "psc_codes": ["D302"]
    },
    ...
  ]
}
```

## 7. Success Metrics & Measurement

### 7.1. Primary Metrics (출시 후 4주 측정)

**목표 1: Weekly Picks 이탈률 감소**
- 현재: 10% 유저가 "할 공고 없다"고 피드백
- 목표: 5% 이하로 감소
- 측정: 
  - Weekly Picks 방문 유저 중 Sequence 생성 없이 이탈한 비율
  - 검색 기능 사용 후 이탈률 vs 미사용 이탈률 비교

**목표 2: 검색을 통한 Sequence 생성**
- 목표: 주간 50건 이상 (전체 Sequence 생성의 약 10%)
- 측정:
  - 검색 결과에서 클릭한 공고 → Sequence 생성 전환율
  - 검색 사용자의 Sequence 생성 수

### 7.2. Secondary Metrics

**검색 기능 사용률:**
- Weekly Picks 방문 유저 중 검색 기능 사용 비율
- 목표: 20% 이상 (10% 이탈 유저 + α)

**검색 성공률:**
- 검색 시도 중 매칭 결과 1개 이상 나온 비율
- 목표: 60% 이상

**재검색률:**
- 1차 검색 후 2차 검색 시도 비율
- 목표: 30% 이상 (1차 실패 시 재시도 유도 성공)

- 검색창 노출 조건: 스크롤 vs 클릭 vs 시간 → 스크롤 기반 선택

**미결정 사항:**
- [ ] 검색창 정확한 UI 디자인 (Sticky header vs In-list CTA)
- [ ] Loading indicator 필요 여부 (검색 속도에 따라 결정)
- [ ] 검색 히스토리 저장 여부 (향후 개인화 가능성)
