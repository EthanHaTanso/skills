## 1. 기능 개요

### 배경

유저는 온보딩 과정에서 "Generate Profile"을 완료한 후, 본인의 Company Description과 매칭된 Opportunity 리스트를 확인합니다. 이 단계에서 유저는:
- 시스템이 자동 생성한 Company Description (Summary + Business Portfolio)을 보유
- 매칭된 Opportunity 리스트를 처음 마주함
- 각 Opportunity가 본인에게 적합한지 판단해야 함

**현재 문제**: 유저가 공고 하나당 3-5분을 투자해야 적합 여부를 판단할 수 있어, [리스트 확인] → [Opportunity 선택] 전환율이 50%에 머물고 있습니다.

**이번 기능의 역할**: 유저가 빠르게 "이 공고가 나와 관련있다"를 인지하고, "어떤 부분이 연결되는지" 맥락을 파악할 수 있도록 **매칭 태그**와 **본문 하이라이트**를 제공합니다.

### 핵심 가치

- **판단 속도 향상**: 3-5분 → 30초 내 1차 스크리닝 가능하도록 시각적 단서 제공
- **판단 정확도 향상**: Semantic 매칭으로 유저가 놓칠 수 있는 관련 내용 발견 지원
- **인지 부하 감소**: "왜 이 공고가 추천됐는지" 명시적으로 보여줌으로써 불확실성 제거
- **온보딩 성공률 향상**: 1회성 온보딩 과정에서 가치를 즉시 전달하여 이탈 방지

## 2. 유저 상황과 기대

### 유저가 이 단계에 도달했을 때의 심리

- "이 많은 공고 중에 나한테 맞는 게 뭐지? 어디서부터 봐야 하지?"
- "Title만 봐서는 잘 모르겠는데, 하나하나 다 읽어야 하나?"
- "왜 이 공고가 나한테 추천된 거지? 연결고리를 모르겠네"
- "시간이 없는데, 빨리 적합한 것만 골라내고 싶다"

### 유저에게 제공해야 할 경험

1. **즉각적인 관련성 인지** — 카드를 보는 순간 "아, 이건 내 Cloud 사업이랑 관련있구나" 파악
2. **맥락적 이해** — "Description의 이 부분이 내 Business Portfolio와 연결된다" 확인
3. **효율적인 스크리닝** — 전체 Description을 읽지 않고도 관련 공고 선별
4. **신뢰감 형성** — 시스템 추천에 대한 이유를 명확히 이해하여 의사결정 자신감 향상

## 3. 주요 UX 원칙

### 3.1. 스캔 우선 (Scan-First)

- 유저가 리스트를 빠르게 훑을 때, **태그만 봐도** 관련성을 즉시 판단할 수 있어야 함
- 태그는 카드 상단에 위치하여 스크롤 시 가장 먼저 눈에 들어옴
- **의도**: 10개 이상의 공고를 확인해야 하는 유저가 5-10초 내 1차 필터링 가능

### 3.2. 점진적 깊이 (Progressive Depth)

| 레벨 | 요소 | 시간 | 목적 |
|------|------|------|------|
| **Level 1** | 매칭 태그 | 5-10초 | "관련있다/없다" 빠른 판단 |
| **Level 2** | 하이라이트된 문장 | 30초-1분 | "어떤 맥락에서 관련있는지" 이해 |
| **Level 3** | 전체 Description | 3-5분 | 상세 검토 및 최종 판단 |

- **의도**: 유저가 필요한 깊이까지만 탐색하고, 불필요한 시간 투자 방지

### 3.3. 노이즈 최소화 (Signal over Noise)

- 하이라이트는 Description의 **20% 이하**로 제한
- 매칭 신뢰도가 낮은 경우 태그/하이라이트를 표시하지 않음 (over-matching 방지)
- **의도**: 하이라이트가 많으면 오히려 가독성이 떨어지고 핵심을 놓침

### 3.4. 맥락 연결 (Context Connection)

- 태그와 하이라이트는 서로 연결된 정보
- 태그 `Cloud Computing`이 있으면, Description 내 해당 매칭 문장이 하이라이트됨
- **의도**: 유저가 "왜 이 태그가 붙었는지" 즉시 확인 가능

## 4. 상세 유저 플로우

### 4.1. 리스트 스캔 — 빠른 관련성 파악

**상황**
- 유저가 Generate Profile 완료 후 Opportunity 리스트 화면에 진입
- Desktop 3개 / Mobile 1개의 카드가 한 화면에 보임
- 유저 심리: "어떤 것부터 봐야 하지?"

**플로우**
1. 유저가 첫 번째 Opportunity 카드를 봄:
   - 카드 상단에 **매칭 태그** 표시 (예: `Cloud Computing`, `AI`)
   - 태그가 눈에 들어오며 "아, 내 Cloud 사업이랑 관련있구나" 인지
2. 유저가 스크롤하며 다른 카드들을 스캔:
   - 각 카드의 태그를 확인하며 관심 공고 1차 선별
   - 태그가 없는 카드는 "내 Business Portfolio와 직접 매칭되는 건 아니구나" 판단
3. 관심 가는 카드 발견 시:
   - Level 2 (하이라이트 확인)으로 진입

**디자인 의도**
- 스크롤만으로도 관련 공고를 빠르게 식별할 수 있도록 태그를 최상단에 배치
- 태그의 시각적 강조로 Title보다 먼저 눈에 들어오게 함

---

### 4.2. 맥락 확인 — 하이라이트 검토

**상황**
- 유저가 태그를 보고 관심있는 공고 발견
- "어떤 부분이 내 사업과 연결되는 거지?" 궁금함
- 전체 Description을 다 읽기 전에 핵심 연결점을 확인하고 싶음

**플로우**
1. 유저가 Description 영역을 봄:
   - 매칭되는 문장이 **노란색 하이라이트**로 표시됨
   - 예: "The contractor shall provide `cloud-based infrastructure modernization` services..."
2. 유저가 하이라이트된 부분을 중심으로 빠르게 읽음:
   - 주변 맥락을 함께 파악하며 실제 관련성 판단
3. 관련성 확신 시:
   - "Interested" 선택
   - 또는 더 상세히 알고 싶으면 전체 Description 읽기

**디자인 의도**
- 하이라이트가 "읽어야 할 부분"을 명확히 가이드
- 유저가 전체 텍스트를 스캔하지 않아도 핵심 연결점 파악 가능

---

### 4.3. 복수 태그 케이스 — 다중 매칭 처리

**상황**
- 유저의 Business Portfolio에 여러 아이템이 있음 (예: Cloud Computing, AI, Cybersecurity)
- 하나의 Opportunity가 여러 Portfolio 아이템과 매칭될 수 있음

**플로우**
1. 카드에 복수의 태그 표시:
   - 예: `Cloud Computing` `AI` `Cybersecurity`
   - 태그는 가로로 나열 (줄바꿈 허용)
2. Description 내 각 태그에 해당하는 문장이 모두 하이라이트:
   - 하이라이트 총량은 20% 이하 유지
   - 20% 초과 시 관련도 높은 순으로 우선 표시
3. 유저가 여러 하이라이트를 통해 "이 공고는 내 여러 사업 영역과 관련있구나" 인지

**디자인 의도**
- 복수 매칭은 해당 공고의 적합도가 높다는 신호
- 단, 하이라이트 과다 방지를 위해 상한선 적용

---

### 4.4. 매칭 없음 케이스 — 태그 미표시 처리

**상황**
- Opportunity가 유저의 Business Portfolio와 Semantic 매칭되지 않음
- 또는 매칭 신뢰도가 기준 미달

**플로우**
1. 카드에 태그 영역이 비어있거나 표시되지 않음
2. Description에 하이라이트 없이 기존 방식대로 표시
3. 유저가 필요 시 전체 Description을 직접 읽어 판단

**디자인 의도**
- Over-matching보다 Under-matching이 신뢰도에 유리
- 매칭이 없다고 해서 공고가 리스트에서 제외되지는 않음 (Relevant Score 기반 정렬 유지)

---

### 4.5. Semantic 매칭 — 다른 표현 연결

**상황**
- 유저의 Business Portfolio: "Cloud Computing"
- Opportunity Description: "data center modernization and infrastructure migration"
- 동일 키워드는 없지만 의미상 연결됨

**플로우**
1. 시스템이 Semantic 매칭으로 연결 판단:
   - "Cloud Computing" ↔ "data center modernization" (High 신뢰도)
   - "Cloud Computing" ↔ "infrastructure migration" (Medium 신뢰도)
2. 카드에 `Cloud Computing` 태그 표시
3. Description에서 "data center modernization" 문장 하이라이트
4. 유저가 "아, 클라우드랑 직접 쓰진 않았지만 관련 내용이구나" 인지

**디자인 의도**
- 키워드 매칭의 한계(Semantic Gap) 극복
- 유저가 놓칠 수 있는 관련 공고 발견 지원

## 5. 엣지 케이스 및 예외 처리

### 5.1. 하이라이트 과다

- **상황**: 매칭되는 문장이 Description의 20%를 초과
- **처리**: 관련도 점수가 높은 순으로 20%까지만 하이라이트
- **복구 옵션**: 없음 (시스템이 자동으로 조절)

### 5.2. 태그 영역 오버플로우

- **상황**: 매칭된 Portfolio 아이템이 많아 태그가 카드 너비를 초과
- **처리**:
  - 첫 줄에 최대한 표시
  - 넘치면 다음 줄로 줄바꿈
  - 또는 "+N more" 형태로 축약 (디자인 검토 필요)

### 5.3. Company Description 미생성

- **상황**: 웹검색 실패 등으로 Company Description이 없는 경우
- **처리**: 매칭 태그/하이라이트 기능 비활성화, 기존 UI 유지
- **사용자 메시지**: 별도 안내 없음 (기능이 없는 것처럼 표시)

### 5.4. Opportunity Description 누락

- **상황**: 특정 Opportunity의 Description이 비어있거나 매우 짧은 경우
- **처리**: 하이라이트 없이 태그만 표시 (가능한 경우), 둘 다 불가능하면 미표시

## 6. 향후 고려사항

### 6.1. 태그 클릭 시 해당 하이라이트로 스크롤

- **현재**: 태그와 하이라이트가 시각적으로만 연결
- **향후**: 태그 클릭 시 해당 하이라이트 위치로 자동 스크롤하여 탐색 효율 향상

### 6.2. 매칭 품질 피드백 수집

- **현재**: 매칭 품질을 시스템이 자체 판단
- **향후**: 유저가 "이 매칭이 맞다/틀리다" 피드백 제공 → 매칭 알고리즘 개선에 활용

### 6.3. 하이라이트 색상 커스터마이징

- **현재**: 노란색 고정
- **향후**: 다크모드 대응, 색맹 사용자 고려한 색상 옵션

### 6.4. Weekly Picks 확장 적용

- **현재 스코프**: 온보딩 Opportunity 리스트
- **향후**: Weekly Picks 화면에도 동일 기능 적용 검토

## 7. 핵심 메트릭

| 메트릭 | 설명 | 목표 |
|--------|------|------|
| **전환율** | [리스트 확인] → [Opportunity 선택] | 50% → **80%** |
| **선택 개수** | 유저당 평균 선택 Opportunity 수 | 모니터링 (증가 기대) |
| **체류 시간** | 리스트 화면 평균 체류 시간 | 모니터링 (감소 기대) |
| **태그 있는 공고 선택률** | 태그가 있는 공고 중 선택된 비율 | 태그 없는 공고 대비 높아야 함 |

## 8. 정책 요약

1. **매칭 태그**: Business Portfolio 아이템명을 Opportunity 카드 상단에 표시하여 빠른 스캔 지원
2. **본문 하이라이트**: Description 내 매칭 문장에 노란색 배경 적용, 상한 20%
3. **Semantic 매칭**: 키워드 일치가 아닌 의미 기반 매칭으로 Semantic Gap 극복
4. **품질 우선**: 매칭 신뢰도가 낮으면 표시하지 않음 (over-matching 방지)
5. **태그 없는 공고**: 리스트에서 제외하지 않고 기존 방식으로 표시
6. **하이라이트 과다 방지**: 20% 초과 시 관련도 높은 순으로 제한
