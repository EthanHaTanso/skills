# MVP Go/No-Go Criteria (판단 기준)

이 문서는 인터뷰 후 내부 검토를 통해 MVP 개발 여부를 결정하는 기준을 제공합니다.

## 판단 프로세스

```
[인터뷰 완료] → [JTBD 문서 작성] → [Go/No-Go 판단] → [GO: MVP 개발 / NO-GO: 다음 기회]
```

---

## 필수 기준: AI 재설계 가능성 (Must-Have)

이 기준을 통과하지 못하면 자동 NO-GO입니다.

### 1. 단계 압축 가능성

**질문**: "여러 단계를 하나로 합칠 수 있는가?"

#### ✅ PASS 예시

**Case 1**: 데이터 수집 + 정제 + 분석 + 리포팅 (4단계) → AI가 자동 분석 + 리포트 생성 (1단계)

**Case 2**: 이미지 다운로드 + 수동 텍스트 추출 + 스프레드시트 입력 (3단계) → 이미지 업로드하면 자동 추출 및 입력 (1단계)

#### ❌ FAIL 예시

**Case 1**: 단순 복사-붙여넣기 자동화 (단계는 그대로, 속도만 빨라짐)

**Case 2**: 기존 Excel 수식을 Python 스크립트로 바꾸기 (도구만 변경, 워크플로우는 동일)

---

### 2. 새로운 가능성 제공

**질문**: "AI로 기존에 불가능했던 것이 가능해지는가?"

#### ✅ PASS 예시

**Case 1**: 자연어로 질문하면 자동으로 데이터 분석 결과 제공
- **이전**: SQL 작성 → 쿼리 실행 → 수동 분석
- **AI 재설계**: "지난 주 가장 많이 팔린 제품은?" → 즉시 답변

**Case 2**: 회의록 자동 생성 + 액션 아이템 자동 추출
- **이전**: 회의 중 수동 메모 → 회의 후 정리 → 액션 아이템 별도 추출
- **AI 재설계**: 회의 녹음 → 자동 회의록 + 액션 아이템

**Case 3**: 이미지에서 구조화된 데이터 자동 추출
- **이전**: 이미지를 보고 수동으로 타이핑
- **AI 재설계**: 이미지 업로드 → JSON/CSV 자동 생성

#### ❌ FAIL 예시

**Case 1**: 버튼 클릭만 자동화 (RPA 수준)
- 기존에 불가능했던 게 아니라, 그냥 귀찮았던 것

**Case 2**: 이미 있는 도구의 재구현
- "Zapier로도 할 수 있는데 직접 만들기"

---

### 3. "못 돌아가는" Value Impact

**질문**: "한 번 써보면 이전 방식으로 돌아갈 수 없을 정도의 임팩트인가?"

#### Impact Level 정의

**High (✅ PASS)**:
- 이전 방식으로 돌아가는 건 상상할 수 없음
- 예시:
  - 구글 검색 없이 정보 찾기
  - 스마트폰 없이 살기
  - GitHub Copilot 없이 코딩하기

**Medium (⚠️ 조건부 PASS)**:
- 훨씬 편하지만 이전 방식도 가능은 함
- 다른 조건(기술적 실현 가능성, Big Fan 가능성)이 매우 좋으면 PASS
- 예시:
  - 자동 완성 기능
  - 단축키
  - 템플릿

**Low (❌ FAIL)**:
- 약간의 개선 수준, 없어도 크게 불편하지 않음
- 예시:
  - UI 색상 변경
  - 알림 타이밍 조정

## 보조 기준: 기술적 실현 가능성

AI 재설계 가능성을 통과한 경우에만 평가합니다.

### 1. 구현 가능 시간

**질문**: "1-2주 내에 구현 가능한가?"

#### 시간 추정 가이드

| 요소 | 시간 |
|------|------|
| 간단한 프론트엔드 (React/Next.js) | 1-2일 |
| AI API 연동 (OpenAI, Claude, etc.) | 1일 |
| 데이터 처리 로직 | 1-2일 |
| 외부 API 연동 (Notion, Slack, etc.) | 1-2일 |
| 배포 (Vercel, Netlify) | 0.5일 |
| 테스트 및 버그 수정 | 1-2일 |

**총 1주일 이내 가능한 예시**:
- 프론트엔드 + AI API + 배포 = 3-4일
- 스크립트 + AI API = 2-3일
- No-code (Zapier) + AI API = 1-2일

#### ✅ PASS 예시
- 간단한 웹 인터페이스 + OpenAI API + 기본 데이터 처리

#### ❌ FAIL 예시
- 복잡한 머신러닝 모델 학습 필요
- 여러 레거시 시스템과의 깊은 통합 필요
- 실시간 처리 및 고성능 최적화 필요

---

### 2. 데이터 접근 가능성

**질문**: "필요한 데이터에 접근 가능한가?"

#### ✅ PASS 예시
- CSV/Excel 파일
- Public API
- 사용자가 직접 제공하는 데이터
- OAuth로 접근 가능한 SaaS (Google, Notion, Slack, etc.)

#### ❌ FAIL 예시
- 사내 DB (VPN 없이 접근 불가)
- API가 없는 레거시 시스템
- 접근 권한을 받을 수 없는 데이터

---

### 3. 기술적 제약 해결 가능성

**질문**: "보안/정책 제약을 해결할 수 있는가?"

#### 제약 유형별 해결 방법

| 제약 | 해결 방법 |
|------|----------|
| 외부 서비스 사용 불가 | → Local App 또는 On-premise 배포 |
| 데이터 외부 전송 불가 | → 로컬에서만 처리, 또는 On-device AI |
| IT 승인 필요 | → 사용자가 승인 받을 수 있는지 확인 |
| 비용 문제 | → 무료 tier로 해결 가능한지 확인 |

#### ✅ PASS 예시
- 보안 제약이 있지만 Local App으로 해결 가능
- IT 승인이 필요하지만 사용자가 받을 수 있음

#### ❌ FAIL 예시
- 보안 제약이 있고 해결 방법이 없음
- IT 승인이 불가능

---

## 제외 기준: 비즈니스 임팩트 (초기 단계에서는 무시)

> **중요**: Closed network는 학습 및 포트폴리오 구축 용도이므로,
> Global Scale 가능성은 평가하지 않습니다.

### 나중에 평가할 항목들

**Indirect/Public 확장 시 평가**:
- 직무 보편성 (해당 직무에서 얼마나 보편적인 워크플로우인가?)
- 네트워크 효과 (인터뷰이의 네트워크 크기, 파급력)
- Big Fan 가능성 (주변에 자랑하고 싶을 만한가?)

**현재는 고려하지 않음**:
- "이 직무가 글로벌로 많은가?"
- "시장 크기는 얼마나 되는가?"
- "수익화 가능성은?"

## 판단 Decision Tree

```
┌─────────────────────────────────────────┐
│ 1. AI 재설계 가능성 평가                │
│    - 단계 압축 가능?                    │
│    - 새로운 가능성 제공?                │
│    - Value Impact: High/Medium?         │
└──────────────┬──────────────────────────┘
               │
               ├─ NO → ❌ NO-GO
               │
               ├─ YES → Continue
               │
┌──────────────▼──────────────────────────┐
│ 2. 기술적 실현 가능성 평가              │
│    - 1-2주 내 구현 가능?                │
│    - 데이터 접근 가능?                  │
│    - 기술적 제약 해결 가능?             │
└──────────────┬──────────────────────────┘
               │
               ├─ NO → ❌ NO-GO
               │
               ├─ YES → ✅ GO!
               │
┌──────────────▼──────────────────────────┐
│ 3. MVP 개발 시작                        │
│    - Scope 정의                         │
│    - Timeline 설정                      │
│    - Demo 일정 잡기                     │
└─────────────────────────────────────────┘
```

## 판단 문서화

Go/No-Go 판단은 JTBD 문서의 "9. MVP Scope" 섹션에 기록합니다.

### GO 판단 예시

```markdown
## 9. MVP Scope (Go/No-Go 판단)

### Go/No-Go Decision
- [x] **GO** - MVP 개발 진행

### 판단 기준 체크리스트

#### AI 재설계 가능성 (필수)
- [x] 단계 압축 가능: 4단계 → 1단계
- [x] AI로 새로운 가능성 제공: 자연어 질의 가능
- [x] "못 돌아가는" Value Impact: High

#### 기술적 실현 가능성
- [x] 1주일 내 구현 가능: React + OpenAI API + Vercel
- [x] 필요한 데이터 접근 가능: CSV 파일 업로드
- [x] 기술적 제약 해결 가능: 제약 없음

### 판단 근거
현재 4단계(데이터 수집, 정제, 분석, 리포팅)를 거쳐 2시간이 걸리는 작업을,
AI로 "지난 주 성과를 요약해줘"라고 질문하면 5분 내 완료 가능.
기존 방식으로 돌아가는 건 상상할 수 없을 정도의 임팩트 예상.
기술적으로도 1주일 내 충분히 구현 가능.
```

### NO-GO 판단 예시

```markdown
## 9. MVP Scope (Go/No-Go 판단)

### Go/No-Go Decision
- [x] **NO-GO** - MVP 개발 안 함

### 판단 기준 체크리스트

#### AI 재설계 가능성 (필수)
- [ ] 단계 압축 가능: 단계는 그대로, 속도만 빨라짐
- [ ] AI로 새로운 가능성 제공: 기존에도 자동화 가능했던 작업
- [ ] "못 돌아가는" Value Impact: Low

### 판단 근거
요청한 작업은 단순 복사-붙여넣기 자동화로, AI가 아니어도
RPA나 간단한 스크립트로 해결 가능.
AI 재설계의 본질적 가치가 없어 NO-GO로 판단.

### 대안 제안
간단한 Python 스크립트로 해결 가능하므로, 인터뷰이에게
스크립트 코드와 사용 방법을 안내하는 것으로 대체.
```

## Edge Cases 처리

### Case 1: AI 재설계는 좋은데 기술적으로 어려운 경우

**판단**: NO-GO
**이유**: 1-2주 내 구현 불가능하면 MVP 범위를 벗어남
**대안**: Scope를 줄여서 재평가

### Case 2: 기술적으로 쉬운데 AI 재설계 가치가 애매한 경우

**판단**: NO-GO
**이유**: 대원칙 위배 - "단순 효율화가 아닌 AI 재설계"
**대안**: 인터뷰이와 다른 pain point 논의

### Case 3: Medium Impact인데 다른 조건은 모두 좋은 경우

**판단**: 조건부 GO
**조건**: 인터뷰이가 Big Fan이 될 가능성이 높은 경우
**예시**: 직무가 명확하고, 해당 직무의 보편적 문제인 경우

## 판단 후 액션

### GO 판단 시

1. **MVP Scope 정의**:
   - In Scope / Out of Scope 명확히
   - 1-2주 내 구현 가능한 범위로 제한

2. **Timeline 설정**:
   - Target Demo Date 설정 (1주일 후)
   - 개발 시작

3. **인터뷰이에게 연락**:
   - "개발 시작했습니다" 메시지
   - Demo 일정 사전 공유

### NO-GO 판단 시

1. **Learning 기록**:
   - 왜 NO-GO인지 문서화
   - 비슷한 케이스 재발 방지

2. **인터뷰이에게 피드백**:
   - 정중하게 이유 설명
   - 가능하면 대안 제안 (스크립트, 기존 도구 추천 등)

3. **다음 인터뷰이 찾기**:
   - 다른 워크플로우 탐색
   - 다른 인터뷰이 모집

## 판단 기준 진화

이 기준은 고정된 것이 아니며, 경험이 쌓이면서 업데이트됩니다.

### 업데이트 시점

- **Big Fan 발견**: 어떤 케이스에서 Big Fan이 나왔는지 패턴 분석
- **실패 케이스**: NO-GO 판단이 맞았는지, 놓친 기회는 없었는지 회고
- **MVP 개발 경험**: 실제로 1주일 내 가능했는지, 어려웠던 점은?

### 학습 루프

```
[인터뷰] → [Go/No-Go] → [MVP 개발] → [Demo] → [폴로업] → [Learning]
                                                              ↓
                                                    [기준 업데이트]
```
