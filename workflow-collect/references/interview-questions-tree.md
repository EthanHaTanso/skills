# Interview Questions Tree (상황별 질문 트리)

이 문서는 인터뷰 중 실시간으로 활용할 수 있는 상황별 질문 트리를 제공합니다.
인터뷰이의 답변에 따라 다음 질문을 분기하면서 깊이 있게 워크플로우를 파악합니다.

## 핵심 원칙

1. **Pain Point 중심 탐색**: 문제보다는 일상 업무에서 시작
2. **구체적 시나리오**: "어제 하루" 또는 "지난 주" 등 구체적 사례
3. **화면 공유 권장**: 실제 작업 화면을 보면서 인터뷰 진행
4. **깊이 파고들기**: 표면적 답변에서 멈추지 말고 "왜?" 질문 반복

---

## Opening (5분)

### 1-1. 시작 질문

**Q**: "어제(또는 오늘) 하루 업무 중 가장 짜증났던 순간이 언제였나요?"

**답변 유형별 분기**:

| 답변 유형 | 다음 질문 |
|----------|----------|
| 구체적 작업 언급 (예: "보고서 작성") | → **1-2-A**로 이동 |
| 추상적 불편 (예: "커뮤니케이션이 힘들어요") | → **1-2-B**로 이동 |
| "특별히 없었어요" | → **1-2-C**로 이동 |

---

### 1-2-A. 구체적 작업이 언급된 경우

**Q**: "그 작업을 지금 화면으로 보여주실 수 있나요? 어떻게 하시는지 보고 싶어요."

**화면 공유 시**:
→ **2. AS-IS Workflow 파악**으로 이동

**화면 공유 불가 시**:
**Q**: "괜찮습니다. 그럼 그 작업을 단계별로 설명해주실 수 있나요? 첫 번째 단계부터요."
→ **2. AS-IS Workflow 파악**으로 이동

---

### 1-2-B. 추상적 불편이 언급된 경우

**Q**: "구체적으로 어떤 상황에서 [언급한 불편]을 느끼셨나요? 최근 사례를 하나 들어주실 수 있나요?"

**구체적 사례 나옴**:
→ **1-2-A**로 이동

**여전히 추상적**:
**Q**: "예를 들어, 어제 오전에는 어떤 작업을 하셨나요?"
→ 일정/작업 목록 중 하나를 선택하도록 유도
→ **1-2-A**로 이동

---

### 1-2-C. "특별히 없었어요" 응답

**Q**: "평소 업무 중 반복적으로 하는 작업이 있나요? 매일 또는 매주 하는 것들이요."

**답변 나옴**:
**Q**: "그 중에서 가장 시간이 오래 걸리거나 귀찮은 건 무엇인가요?"
→ **1-2-A**로 이동

**여전히 막막**:
**Q**: "그럼 이렇게 접근해볼까요. 어제 하루 일정을 쭉 떠올려보시면, 어떤 작업들을 하셨나요?"
→ 일정을 하나씩 나열하도록 유도
→ 그 중 하나 선택하여 **1-2-A**로 이동

---

## AS-IS Workflow 파악 (10-15분)

### 2-1. 단계별 분해

**Q**: "이 작업을 처음부터 끝까지 단계별로 설명해주시겠어요?"

**인터뷰이가 단계 설명 시작**:
- 각 단계마다 메모
- 불분명한 부분은 즉시 질문

**각 단계별 추가 질문**:

---

### 2-2. 각 단계 Deep Dive

인터뷰이가 언급한 각 단계마다 다음을 확인:

#### Step N 확인 질문

**Q**: "[Step N]에서는 구체적으로 무엇을 하시나요?"

**답변 유형별 분기**:

| 답변 유형 | 다음 질문 |
|----------|----------|
| 수동 입력/복붙 (예: "데이터 복사해서 붙여넣어요") | → **2-3-A. 수동 작업 탐색** |
| 도구 사용 (예: "엑셀로 계산해요") | → **2-3-B. 도구 사용 탐색** |
| 대기/확인 (예: "결과를 기다려요") | → **2-3-C. 대기 시간 탐색** |
| 판단/결정 (예: "이게 맞는지 확인해요") | → **2-3-D. 판단 프로세스 탐색** |

---

### 2-3-A. 수동 작업 탐색

**Q**: "이 작업을 하는 데 보통 얼마나 걸리시나요?"

**시간이 긴 경우 (10분 이상)**:
**Q**: "매번 이렇게 하시나요? 얼마나 자주 하시나요?"

**빈도 확인**:
- 매일? → **High Priority Pain Point**
- 매주? → **Medium Priority**
- 가끔? → **Low Priority**

**Q**: "이 과정에서 실수가 나거나 오류가 생기는 경우도 있나요?"

**실수 있음**:
**Q**: "어떤 종류의 실수가 많나요? 실수하면 어떻게 되나요?"
→ **3. Pain Point 확인**으로 이동

---

### 2-3-B. 도구 사용 탐색

**Q**: "왜 [도구 이름]을 사용하시나요? 다른 도구를 써보신 적 있나요?"

**답변 분석**:
- "회사에서 쓰라고 해서" → 강제 사용 (전환 어려움)
- "편해서" → 습관 (개선 여지 있음)
- "이것밖에 몰라서" → 교육 필요 (개선 여지 큼)

**Q**: "[도구]를 사용할 때 불편한 점은 없나요?"

**불편한 점 나옴**:
→ **3. Pain Point 확인**으로 이동

---

### 2-3-C. 대기 시간 탐색

**Q**: "기다리는 동안 다른 작업을 하시나요, 아니면 그냥 기다리시나요?"

**그냥 기다림**:
**Q**: "보통 얼마나 기다리시나요?"
→ 대기 시간이 10분 이상이면 **High Priority Pain Point**

**다른 작업 함**:
**Q**: "기다리는 동안 다른 작업을 하다가, 결과 나온 걸 어떻게 확인하시나요?"
→ 멀티태스킹 오버헤드 확인

---

### 2-3-D. 판단 프로세스 탐색

**Q**: "어떤 기준으로 판단하시나요?"

**명확한 기준 있음**:
**Q**: "그 기준이 문서화되어 있나요? 아니면 경험으로 아시는 건가요?"

**기준 애매함**:
**Q**: "판단이 어려운 경우도 있나요? 그럴 때는 어떻게 하시나요?"
→ **3. Pain Point 확인**으로 이동

---

## Pain Point 확인 (5-10분)

### 3-1. Pain Point 명확화

**Q**: "지금까지 말씀하신 작업에서, 가장 개선하고 싶은 부분은 무엇인가요?"

**답변 분석**:

| Pain Point 유형 | 다음 질문 |
|-----------------|----------|
| 시간 소모 | → **3-2-A** |
| 오류 발생 | → **3-2-B** |
| 지루함/반복 | → **3-2-C** |
| 복잡함 | → **3-2-D** |

---

### 3-2-A. 시간 소모 Pain Point

**Q**: "이 작업에 시간을 얼마나 쓰시나요? (하루/주/월 기준)"

**시간 계산**:
- 하루 30분 이상? → **High Impact**
- 주당 2시간 이상? → **High Impact**

**Q**: "만약 이 시간을 절약할 수 있다면, 그 시간에 무엇을 하고 싶으신가요?"
→ Desired Outcome 파악

→ **4. Desired State 파악**으로 이동

---

### 3-2-B. 오류 발생 Pain Point

**Q**: "오류가 얼마나 자주 나나요?"

**빈도 확인**:
- 매번? → **Critical Issue**
- 가끔? → 어떤 경우에 나는지 패턴 파악

**Q**: "오류가 나면 어떻게 되나요? 수정하는 데 시간이 얼마나 걸리나요?"

**Impact 분석**:
- 고객/상사에게 영향? → **High Impact**
- 혼자 수정 가능? → **Medium Impact**

→ **4. Desired State 파악**으로 이동

---

### 3-2-C. 지루함/반복 Pain Point

**Q**: "이 작업을 하면서 '이거 자동화되면 좋겠다'고 생각해보신 적 있나요?"

**시도해본 적 있음**:
**Q**: "어떤 방법을 시도해보셨나요? 왜 안 됐나요?"
→ 기술적 장벽 파악

**시도 안 해봄**:
**Q**: "시도해보지 않으신 이유가 있나요?"
→ 시간 부족? 기술 부족? 우선순위 낮음?

→ **4. Desired State 파악**으로 이동

---

### 3-2-D. 복잡함 Pain Point

**Q**: "이 작업이 왜 복잡하다고 느끼시나요?"

**답변 분석**:
- 단계가 많음 → 단계 압축 가능성
- 예외 케이스 많음 → AI로 처리 가능성
- 판단이 어려움 → AI 보조 가능성

**Q**: "이 작업을 처음 배울 때 얼마나 걸렸나요? 새로운 사람에게 설명하기 어려운가요?"
→ Onboarding 오버헤드 확인

→ **4. Desired State 파악**으로 이동

---

## Desired State 파악 (5-10분)

### 4-1. 이상적인 상태 질문

**Q**: "만약 마법처럼 이 작업이 자동화된다면, 어떤 모습이면 좋겠어요?"

**인터뷰이가 구체적 묘사**:
→ 메모하고 **4-2**로 이동

**인터뷰이가 막막해함**:
**Q**: "예를 들어, '버튼 하나만 누르면 끝'이면 좋겠다는 건가요? 아니면 '이 부분만이라도 자동이면 좋겠다'는 건가요?"
→ 구체화 유도

---

### 4-2. 제약 조건 파악

**Q**: "회사에서 외부 도구나 서비스를 사용하는 데 제약이 있나요?"

**제약 유형 확인**:

| 제약 유형 | 다음 질문 |
|----------|----------|
| 보안 정책 (외부 접근 불가) | → "로컬에서 실행되는 도구도 괜찮으신가요?" |
| 비용 문제 | → "무료 도구만 가능한가요?" |
| IT 승인 필요 | → "IT 부서 승인 프로세스가 있나요?" |
| 제약 없음 | → **4-3**으로 이동 |

---

### 4-3. 데이터 접근성 파악

**Q**: "이 작업에 필요한 데이터는 어디에 있나요?"

**데이터 위치 확인**:
- Excel/CSV 파일? → 쉬움
- 사내 DB? → API 접근 가능 여부 확인
- 웹사이트? → 스크래핑 가능 여부 확인
- 여러 곳에 분산? → 통합 필요

**Q**: "그 데이터에 접근하는 데 특별한 권한이 필요한가요?"

→ **5. Technical Feasibility 체크**로 이동

---

## Technical Feasibility 체크 (5분)

### 5-1. 현재 도구 스택 확인

**Q**: "현재 어떤 도구들을 주로 사용하시나요?"

**도구 리스트 확인**:
- Excel/Google Sheets?
- Notion/Confluence?
- Slack/Teams?
- Salesforce/HubSpot?
- 기타 SaaS?

→ 각 도구의 API 연동 가능성 체크

---

### 5-2. 기술 수준 파악

**Q**: "혹시 코딩이나 스크립트를 작성해보신 경험이 있으신가요?"

**경험 있음**:
**Q**: "어떤 언어나 도구를 사용하셨나요?"
→ 전달 형태 결정에 참고 (스크립트? 웹앱? No-code?)

**경험 없음**:
**Q**: "그럼 웹 브라우저에서 클릭만으로 사용할 수 있는 게 편하실 것 같은데, 맞나요?"
→ 웹앱 형태 선호

---

### 5-3. MVP 스코프 합의

**Q**: "제가 1주일 안에 만들어드릴 수 있는 범위를 정해야 하는데요, 전체 작업 중 어느 부분을 먼저 자동화하면 가장 도움이 될까요?"

**우선순위 확인**:
1. 가장 시간이 오래 걸리는 부분?
2. 가장 오류가 많이 나는 부분?
3. 가장 지루한 부분?

**Q**: "만약 [우선순위 1]이 자동화된다면, 나머지는 직접 하셔도 괜찮으신가요?"

**스코프 합의 완료**:
→ **인터뷰 종료**

---

## 인터뷰 종료 (2분)

### 마무리 질문

**Q**: "마지막으로, 제가 놓친 부분이나 추가로 말씀하고 싶은 게 있으신가요?"

**감사 인사**:
"오늘 귀중한 시간 내주셔서 감사합니다. 일주일 내로 데모를 준비해서 다시 연락드릴게요!"

**다음 액션**:
1. 인터뷰 내용을 JTBD 문서로 정리
2. Go/No-Go 판단
3. MVP 개발 시작

## 질문 트리 사용 팁

### Tip 1: 유연하게 사용
- 이 트리는 가이드일 뿐, 엄격하게 따를 필요 없음
- 인터뷰이의 답변 흐름에 따라 자연스럽게 진행

### Tip 2: 깊이 파고들기
- 표면적 답변에 만족하지 말고 "왜?"를 3번 이상 물어보기
- 구체적인 사례와 숫자를 요청

### Tip 3: 화면 공유 활용
- 가능하면 화면 공유로 실제 작업을 보기
- 설명보다 실제를 보는 게 훨씬 명확함

### Tip 4: 메모 실시간 작성
- 인터뷰 중 실시간으로 메모 (JTBD 구조에 맞춰)
- 인터뷰 끝나고 기억에 의존하면 놓치는 부분 많음

### Tip 5: 시간 배분
- Opening: 5분
- AS-IS Workflow: 10-15분
- Pain Point 확인: 5-10분
- Desired State: 5-10분
- Technical Feasibility: 5분
- 마무리: 2분
- **총 30-40분**

시간이 부족하면, Technical Feasibility는 인터뷰 후 별도로 확인 가능.
