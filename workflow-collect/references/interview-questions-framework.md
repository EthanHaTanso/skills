# Interview Questions Framework (프레임워크 + 예시)

이 문서는 JTBD 프레임워크를 기반으로 한 인터뷰 질문 구조와 예시를 제공합니다.
상황별 질문 트리와 달리, 이 문서는 JTBD의 각 영역(Job, Context, Pain Point, Outcome)별로
질문을 정리하여 체계적으로 접근할 수 있도록 합니다.

## JTBD Framework Overview

```
┌─────────────────────────────────────────────┐
│ Job: 인터뷰이가 수행하는 핵심 작업          │
├─────────────────────────────────────────────┤
│ Context: 작업이 발생하는 상황/조건          │
├─────────────────────────────────────────────┤
│ AS-IS Flow: 현재 어떻게 작업하는지          │
├─────────────────────────────────────────────┤
│ Pain Points: 현재 프로세스의 문제점         │
├─────────────────────────────────────────────┤
│ Desired Outcome: 원하는 결과/상태           │
├─────────────────────────────────────────────┤
│ Technical Constraints: 기술적 제약 조건     │
└─────────────────────────────────────────────┘
```

---

## 1. Job (핵심 작업 파악)

### 목표
인터뷰이가 "무엇을 하려고" 이 작업을 수행하는지 파악

### 핵심 질문

**Q1**: "주로 어떤 업무를 담당하고 계신가요?"
- **목적**: 전체 직무 컨텍스트 파악
- **예상 답변**: "마케팅 캠페인 성과 분석을 담당합니다", "계약서 검토 업무를 합니다"

**Q2**: "그 중에서 가장 시간을 많이 쓰는 작업은 무엇인가요?"
- **목적**: High-impact 작업 식별
- **예상 답변**: "주간 리포트 작성", "데이터 수집 및 정제"

**Q3**: "그 작업의 최종 목적은 무엇인가요? (누가, 왜 필요로 하나요?)"
- **목적**: 작업의 가치/중요도 파악
- **예상 답변**: "경영진이 의사결정에 사용합니다", "고객에게 제공하는 산출물입니다"

### 직무별 예시 질문

#### 마케터

- "캠페인 성과를 분석할 때, 가장 중요하게 보는 지표는 무엇인가요?"
- "리포트를 누구에게 제출하시나요? 어떤 질문을 많이 받으시나요?"
- "데이터 수집부터 리포트 완성까지 얼마나 걸리나요?"

#### 데이터 분석가

- "분석 요청이 들어오면, 첫 번째로 하는 작업은 무엇인가요?"
- "분석 결과를 어떤 형태로 전달하시나요? (대시보드, 보고서, 슬라이드?)"
- "분석 중 가장 시간이 오래 걸리는 부분은 무엇인가요?"

#### 법무 담당자

- "계약서 검토 시, 주로 어떤 조항을 중점적으로 보시나요?"
- "검토 후 피드백은 어떤 형태로 전달하시나요?"
- "계약서 유형이 여러 개인가요? 각각 검토 프로세스가 다른가요?"

#### 제품 관리자

- "기능 우선순위를 어떻게 결정하시나요?"
- "개발팀과 커뮤니케이션할 때 어떤 문서를 주로 사용하시나요?"
- "유저 피드백을 어떻게 수집하고 정리하시나요?"

## 2. Context (작업 발생 상황/조건)

### 목표
작업이 "언제, 어떤 상황에서" 발생하는지 파악

### 핵심 질문

**Q1**: "이 작업을 얼마나 자주 하시나요?"
- **목적**: 빈도 파악 (매일/매주/매월)
- **예상 답변**: "매주 월요일마다", "캠페인 종료할 때마다"

**Q2**: "이 작업을 하기 전에 보통 무엇을 하시나요? (Trigger 파악)"
- **목적**: 작업 시작 조건 파악
- **예상 답변**: "상사가 요청하면", "시스템에서 알림이 오면"

**Q3**: "이 작업을 하는 데 필요한 정보/데이터는 어디서 얻나요?"
- **목적**: 데이터 소스 파악
- **예상 답변**: "Google Analytics, Salesforce, Excel 파일"

**Q4**: "이 작업을 혼자 하시나요, 아니면 다른 사람과 협업하시나요?"
- **목적**: 협업 여부 파악
- **예상 답변**: "디자이너와 함께", "혼자 합니다"

### 상황별 예시 질문

#### 정기적 작업 (매일/매주)

- "이 작업을 하는 시간대가 정해져 있나요?"
- "마감 시한이 있나요? 급하게 해야 하는 경우도 있나요?"
- "비슷한 작업을 반복하는 건가요, 매번 다른 작업인가요?"

#### 요청 기반 작업 (Ad-hoc)

- "요청이 얼마나 자주 들어오나요?"
- "요청을 받으면 즉시 처리하시나요, 아니면 모아서 처리하시나요?"
- "요청 내용이 명확한가요, 아니면 추가 확인이 필요한 경우가 많나요?"

#### 협업 작업

- "다른 사람의 작업 결과를 기다려야 하나요?"
- "중간에 피드백을 주고받는 과정이 있나요?"
- "최종 산출물을 누가 검토하나요?"

## 3. AS-IS Flow (현재 프로세스)

### 목표
"어떻게" 작업을 수행하는지 단계별로 상세히 파악

### 핵심 질문

**Q1**: "이 작업을 처음부터 끝까지 단계별로 설명해주시겠어요?"
- **목적**: 전체 플로우 파악
- **Tip**: 각 단계마다 메모하고, 불분명한 부분은 즉시 질문

**Q2**: "각 단계별로 얼마나 시간이 걸리시나요?"
- **목적**: 시간 소모가 큰 단계 식별
- **예상 답변**: "데이터 수집: 30분, 정제: 1시간, 분석: 30분"

**Q3**: "이 과정에서 어떤 도구들을 사용하시나요?"
- **목적**: 도구 스택 파악
- **예상 답변**: "Excel, Python, Tableau"

### 단계별 Deep Dive 질문

각 단계마다 다음을 확인:

#### 입력 (Input)

- "이 단계를 시작하기 위해 필요한 것은 무엇인가요?"
- "데이터 형식은 어떤가요? (CSV, JSON, API?)"
- "데이터 품질에 문제가 있는 경우도 있나요?"

#### 처리 (Process)

- "이 단계에서 구체적으로 무엇을 하시나요?"
- "수동으로 하시나요, 아니면 자동화된 부분이 있나요?"
- "예외 케이스가 있나요? 어떻게 처리하시나요?"

#### 출력 (Output)

- "이 단계의 결과물은 무엇인가요?"
- "결과물이 다음 단계에 바로 사용 가능한가요, 아니면 추가 작업이 필요한가요?"
- "결과물을 검증하는 방법이 있나요?"

### 작업 유형별 예시

#### 데이터 처리 작업

1. **수집**: "어떤 소스에서 데이터를 가져오시나요?"
2. **정제**: "데이터를 정제할 때 주로 하는 작업은 무엇인가요? (중복 제거, 결측치 처리 등)"
3. **변환**: "데이터 형식을 바꾸는 작업이 있나요?"
4. **분석**: "어떤 분석을 수행하시나요? (집계, 시각화, 통계 분석?)"
5. **리포팅**: "최종 결과를 어떤 형태로 만드시나요?"

#### 문서 작업

1. **템플릿**: "기존 템플릿을 사용하시나요, 아니면 매번 새로 만드시나요?"
2. **내용 작성**: "어디서 정보를 가져와서 작성하시나요?"
3. **검토**: "작성 후 스스로 검토하시나요? 체크리스트가 있나요?"
4. **승인**: "다른 사람의 승인이 필요한가요?"
5. **배포**: "최종 문서를 어떻게 전달하시나요?"

#### 커뮤니케이션 작업

1. **정보 수집**: "미팅 전에 어떤 준비를 하시나요?"
2. **회의**: "회의록을 작성하시나요?"
3. **Follow-up**: "회의 후 액션 아이템을 어떻게 관리하시나요?"

## 4. Pain Points (문제점)

### 목표
현재 프로세스의 "무엇이 불편한지" 파악

### 핵심 질문

**Q1**: "이 작업에서 가장 짜증나는 부분은 무엇인가요?"
- **목적**: 주관적 불편함 파악
- **예상 답변**: "데이터 복사-붙여넣기가 너무 많아요", "실수하기 쉬워요"

**Q2**: "이 작업 때문에 다른 중요한 일을 못한 적이 있나요?"
- **목적**: 기회비용 파악
- **예상 답변**: "네, 전략 수립 시간이 부족해요"

**Q3**: "실수나 오류가 발생한 적이 있나요? 어떤 종류의 실수였나요?"
- **목적**: 리스크 파악
- **예상 답변**: "숫자를 잘못 입력해서 리포트를 다시 만든 적이 있어요"

### Pain Point 유형별 질문

#### 시간 소모

- "이 작업에 하루/주/월 기준으로 총 얼마나 시간을 쓰시나요?"
- "시간을 가장 많이 잡아먹는 부분은 어디인가요?"
- "만약 이 시간을 절약할 수 있다면, 그 시간에 무엇을 하고 싶으신가요?"

#### 반복/지루함

- "같은 작업을 반복하는 느낌이 드시나요?"
- "이 작업을 하면서 '이거 자동화되면 좋겠다'고 생각해보신 적 있나요?"
- "자동화를 시도해보신 적 있나요? 시도했다면 왜 안 됐나요?"

#### 오류/실수

- "오류가 얼마나 자주 발생하나요?"
- "오류를 발견하는 방법이 있나요, 아니면 나중에 알게 되나요?"
- "오류를 수정하는 데 얼마나 걸리나요?"
- "오류로 인한 손실이나 피해가 있나요?"

#### 복잡함

- "이 작업을 처음 배울 때 얼마나 걸렸나요?"
- "새로운 사람에게 이 작업을 가르치기 어려운가요?"
- "예외 케이스가 많아서 헷갈리시나요?"
- "판단이 애매한 경우가 많나요?"

#### 대기/병목

- "다른 사람의 작업을 기다려야 하나요?"
- "시스템이 느려서 기다리는 시간이 많나요?"
- "승인 프로세스가 오래 걸리나요?"

#### 협업 문제

- "다른 팀과 커뮤니케이션할 때 문제가 있나요?"
- "정보가 여러 곳에 흩어져 있어서 찾기 어려운가요?"
- "버전 관리가 어려운가요? (누가 최신 버전인지 모르는 경우)"

## 5. Desired Outcome (원하는 결과)

### 목표
"무엇이 해결되면" 만족할지 파악

### 핵심 질문

**Q1**: "이상적으로는 이 작업이 어떻게 되면 좋겠어요?"
- **목적**: 이상적 상태 파악
- **예상 답변**: "클릭 한 번에 리포트가 자동 생성되면 좋겠어요"

**Q2**: "이 작업이 개선되면 어떤 점이 가장 좋을 것 같으세요?"
- **목적**: 기대 효과 파악
- **예상 답변**: "시간 절약, 실수 감소, 더 중요한 일에 집중"

**Q3**: "비슷한 문제를 해결하는 도구를 써보신 적 있나요?"
- **목적**: 레퍼런스 파악
- **예상 답변**: "Zapier 써봤는데 우리 케이스에는 안 맞았어요"

### Outcome 유형별 질문

#### 시간 절약

- "이 작업에 걸리는 시간이 절반으로 줄어든다면 충분할까요?"
- "아니면 거의 자동화되어서 5분 내로 끝나야 할까요?"

#### 품질 향상

- "오류율이 0%가 되면 만족하실까요?"
- "아니면 오류가 나더라도 빠르게 발견할 수 있으면 될까요?"

#### 확장성

- "앞으로 이 작업의 양이 늘어날 가능성이 있나요?"
- "지금은 작지만 나중에 팀 전체가 쓸 수도 있나요?"

## 6. Technical Constraints (기술적 제약)

### 목표
"무엇이 가능하고 불가능한지" 파악

### 핵심 질문

**Q1**: "회사에서 외부 도구나 서비스를 사용하는 데 제약이 있나요?"
- **목적**: 보안/정책 제약 파악
- **예상 답변**: "보안 정책 때문에 외부 서비스 사용 어려워요"

**Q2**: "데이터가 외부로 나가는 것에 문제가 있나요?"
- **목적**: 데이터 보안 제약 파악
- **예상 답변**: "고객 데이터라 외부 전송 불가능합니다"

**Q3**: "IT 부서의 승인이 필요한가요?"
- **목적**: 승인 프로세스 파악
- **예상 답변**: "새로운 소프트웨어 설치 시 IT 승인 필요"

### 제약 유형별 질문

#### 보안 제약

- "VPN 없이 외부에서 접근 가능한가요?"
- "클라우드 서비스 사용 가능한가요?"
- "데이터베이스에 직접 접근 가능한가요?"

#### 기술 스택 제약

- "회사에서 특정 프로그래밍 언어나 플랫폼을 강제하나요?"
- "기존 시스템과 통합되어야 하나요?"
- "API가 제공되나요?"

#### 비용 제약

- "유료 서비스 사용 가능한가요?"
- "예산 승인 프로세스가 있나요?"

#### 사용자 기술 수준

- "코딩 경험이 있으신가요?"
- "터미널/커맨드라인 사용 가능하신가요?"
- "웹 브라우저에서만 사용 가능한 게 편하신가요?"

## 프레임워크 사용 가이드

### 사용 시점

이 프레임워크는 다음과 같은 경우에 유용합니다:

1. **체계적인 인터뷰를 원할 때**: 상황별 질문 트리보다 구조화된 접근
2. **JTBD 문서를 실시간으로 작성할 때**: 각 섹션별로 질문하며 바로 문서화
3. **복잡한 워크플로우를 파악할 때**: 여러 단계와 조건이 얽혀 있는 경우

### 인터뷰 흐름 예시

**1. Job 파악 (5분)**
→ "주로 어떤 업무를 하시나요?" 등

**2. Context 파악 (5분)**
→ "얼마나 자주 하시나요?" 등

**3. AS-IS Flow 파악 (10-15분)**
→ "단계별로 설명해주세요" 등

**4. Pain Points 파악 (5-10분)**
→ "가장 짜증나는 부분은?" 등

**5. Desired Outcome 파악 (5분)**
→ "이상적으로는 어떻게 되면 좋겠어요?" 등

**6. Technical Constraints 파악 (5분)**
→ "외부 도구 사용 제약이 있나요?" 등

### 상황별 질문 트리 vs 프레임워크 비교
